## Security

- **[DECISION]** `ps_finding_parse_line` (`extract_str`/`extract_num` via `strstr`) is not a real JSON parser. Fine for self-emitted lines, fragile on malformed input. Replace with a proper parser or fence it off behind input validation.
- **[OPEN]** No fuzzing harness visible for the JSONL ingest path, pcap module parsers, or `agent_proto` frame reader. A tool whose job is parsing untrusted network input should have AFL/libFuzzer on at least the frame reader, the discovery wire format parser, and `ps_finding_parse_line`.
- **[OPEN]** `/register` is unauthenticated by design (validation gate is the trust step) but becomes a flood vector at fleet scale. Add source-IP rate limit + per-IP backoff before anyone notices. `ip_address` field is already captured. — *cheap now, expensive retroactively.*
- **[DECISION]** Replay-much-later not gated: `skew_ms` is computed at central but doesn't appear to reject. If the dedup index ages out, a captured envelope replays cleanly with original `ts`. Decide whether central rejects envelopes with `|skew_ms| > threshold` at ingest, or accepts dedup-at-rest as sufficient.
- **[OPEN]** Engineer-validation step is load-bearing for the entire trust model. `binary_checksum` exists for verification but the operator workflow that actually checks it isn't documented. Define it before fleet scale.
- **[DECISION]** License gov-use exclusion is unenforceable in practice; if the goal is to limit gov adoption, the exclusion does not actually do that. If the goal is something else, name it.

## Code quality

- **[OPEN]** `ps_json` doesn't support object nesting. `tls_probe.c` and `host_table.c` each carry hand-rolled `snprintf` workarounds with comments admitting the smell. Fix the JSON builder rather than work around it a third time. — *note: the reporter/relay envelope builders embed pre-serialized JSON as strings, sidestepping this; but the modules still need nesting.*
- **[OPEN]** Big fixed buffers: `client_hello[4096]`, SMTP recv `buf[8192]`. Bounded but tight; unusual responses truncate evidence. Either size up or document the truncation behavior.
- **[OPEN]** Fixed-size host table (`PS_MAX_HOSTS`) imposes a hard ceiling on observed-host cardinality. Fine for trunk probes, surprising on a busy bridge. Document the limit or switch to a growable structure.
- **[OPEN]** No CI visible. Tests exist but the matrix (which platforms, which compilers, which OpenSSL versions, schedule) isn't apparent. A `.github/workflows/` or equivalent showing macOS + Linux + FreeBSD coverage would prove the portability claim in `build.sh`. — **blocks the parity-test-in-CI item below.**

## Architecture and concept

- **[DECISION]** C across the whole stack defensible short-term, but the memory-safety tax compounds. Worth a deliberate revisit at v2 about whether new code (recipe interpreter, relay forwarder) starts in Rust with a stable C ABI back to libpacketsonde.
- **[OPEN]** Recipe framework is the killer feature and is the furthest out (v3/v4). Until it lands, packetsonde reads as "smaller testssl.sh that emits JSONL." The current signing/registration work is substrate for it (signed request execution is half of what recipes need). Worth wiring that recognition into the relay spec so primitive transport is reserved in the wire format now, not retrofitted later.
- **[OPEN]** Adjacent space (runZero, nuclei, Zeek, Suricata, testssl.sh) is crowded. The differentiator is the deployment-models + recipe-framework combination. The recipe framework being deferred means the differentiation isn't visible yet.

## Event reporting / signing (just landed)

- **[DONE]** Sign-over-bytes (D1) is correctly the headline decision. Commit forwarders to opaque byte forwarding in the relay spec before someone "optimizes" with re-batching or repacking. Any re-encoding breaks the sig. — *Relay Spec A forwards the envelope's payload + ed25519_sig untouched; relays only APPEND to a top-level `relay_path`. Opaque-payload forwarding is locked in.*
- **[PARTIAL]** C↔Python parity test in §10 is the load-bearing test. Cross-language sign/verify is where bugs hide. Make sure this runs in CI, not as a manual integration check. — *test exists (`scripts/test-signing-parity.sh`) but is manual; no CI yet (see Code quality). NEEDS CI.*
- **[PARTIAL/DECISION]** `via_agent` splicing in `audit_via.c` (the legacy v1.6 multi-hop audit path) is incompatible with end-to-end signing. It modifies the payload in flight, which invalidates any inner sig. The relay spec needs to either deprecate this path or move via_agent metadata into a separate signed-by-forwarder attestation alongside the unmodified envelope. — *The new ENVELOPE relay path (Spec A) already does the right thing (attested `relay_path`, payload untouched). But the legacy `--via` AUDIT path still splices `via_agent` into finding frames — that path is separate from envelope reporting and remains incompatible with inner signing. Decide: deprecate the legacy splice, or move it to a forwarder attestation.*

## Future relay phase

- **[DONE]** Decide forwarding model before agents start relaying. Options: opaque byte forwarding (clean, loses chain-of-custody evidence) or attested forwarding (each relay signs a "received from X, forwarding to Y at ts" alongside the original envelope, gives the forensic property the whitepaper claims). The whitepaper's "a finding stream is its own audit trail" claim only holds with the second. — *Spec A chose ATTESTED forwarding exactly as recommended: each relay appends a signed `{relay_agent_id, received_from, ts, sig}` to `relay_path`, central verifies each hop. The forensic property holds.*
- **[OPEN → Spec B]** `authorized_dir/<fpr>.pub` means "this key can do anything." Per-key channel/verb ACLs become important when passive findings flow over the wire. Move to `authorized/<fpr>.toml` with `allowed_kinds` and `allowed_verbs` fields, default-deny on anything not listed. — *Spec A's `allow_sources` is a coarse per-relay list; this is the finer-grained successor, fits Spec B (collector/subscription).*
- **[OPEN → Spec B]** Knock-then-listen vs persistent subscriptions are in tension. Persistent kills the stealth property. Periodic re-knock with a short subscription window requires a bounded ring buffer of recent findings on the agent. Decide retention policy (size? duration? what gets dropped when full?).
- **[OPEN → Spec B]** Backpressure: source-side filtering matters. A subscriber-side filter doesn't help if the agent ships everything to the wire first. Subscription requests need to be expressive enough to push the predicate (kind glob, severity floor) to the agent.
- **[OPEN → Spec B]** Bandwidth on busy bridge with flow tracker + honeypot will be significant. Same problem as above; source filtering is the answer.

## Key management

- **[OPEN]** Key rotation is deferred but needs to land before fleet scale, not after. Checksum-gated rotation (the agent proves identity continuity via binary checksum lineage) is the right shape per the registration spec.
- **[OPEN]** Stolen subscriber key needs to be cuttable without rebuilding everyone's `authorized_dir/`. Some form of central revocation list, or a TTL on authorizations, or both.
- **[OPEN]** No documented key-compromise drill. Operators with a fleet need a runbook for "subscriber X's laptop was lost, what do I do."

## Documentation

- **[OPEN]** The wire contract (envelope shape, `/register` and `/events` endpoint shapes, validation gate semantics, sign-over-bytes rule) is implicit in specs that reference `rna-packetsonde`. Pull it out as `docs/specs/central-protocol-v1.md` so a third-party collector implementation can be written without reading Python source. The toolkit/sink separation the whitepaper promises only works if the contract is documented separately from the reference implementation. — **enables the Spec B collector + any third-party sink; high leverage, low effort.**
- **[OPEN]** Deployment guide should eventually say "point `[central].url` at any endpoint that speaks the v1 contract" rather than implying RNA is required.
- **[OPEN]** Whitepaper "no central finding database, JSONL is the format" survives intact but should note that central is an optional second sink for signed delivery + queryable index, with JSONL/`--auto-append` still the baseline.
- **[OPEN]** Bridge-appliance guide needs disclosure language about what the passive findings bus (once relay lands) ships and to whom. Consider having the agent emit a startup finding declaring which subscriber fingerprints have which subscriptions active.
- **[OPEN]** `ps_json` nesting limitation should be documented as a known issue with the fix planned, not just worked around case by case in module source.

## Operational

- **[OPEN]** The validation UX (`pending → validated` in admin UI) is the trust gate. If it's one click, fine. If it requires reading the binary checksum and out-of-band verifying against what salt deployed, friction kills adoption at scale. Define and document the workflow.
- **[OPEN]** Unauthenticated `/register` becomes a flood vector at scale (see Security). Operationally cheap to fix; expensive to fix retroactively.
